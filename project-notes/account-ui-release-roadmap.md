# Account UI Release Roadmap

## Goal

Enable optional mobile sign-in for a later Play release without making search account-gated or reopening Price Watch and Deep Dive. The user-facing promise is deliberately narrow:

> Sign in to keep saved searches across devices and choose how future shortlists are ranked.

Accounts remain optional. Signed-out users keep device-local history and can complete the full search flow.

## Current starting point

- Account UI, email/password auth, Google OAuth, SecureStore session persistence, local-to-account saved-search migration, ranking preferences, and account deletion are implemented behind `EXPO_PUBLIC_ACCOUNT_UI_ENABLED=true`.
- The production build has this flag off and does not contain the Supabase public configuration.
- The existing Play Data Safety declaration intentionally omits Email address and User IDs for that account-free build. See `project-notes/play-store/review.md`.
- Price Watch and Deep Dive must remain disabled for this release (`EXPO_PUBLIC_PRICE_WATCH_UI_ENABLED` and `EXPO_PUBLIC_DEEP_DIVE_UI_ENABLED` stay false/unset).
- The public deletion page is `https://focamai.com/delete-account`; the app has a signed-in Settings → Account deletion path.

## Non-goals

- Do not gate search, refinement, results, or retailer clickouts behind sign-in.
- Do not add Android dangerous permissions. The existing Android permission list remains unchanged; Google OAuth uses a browser session, not a device permission.
- Do not ship account UI just because the implementation exists. It ships only when the live service and Play declarations are ready.
- Do not turn on Price Watch, Deep Dive, push notifications, or a paid Supabase custom auth domain as part of this work.

## Release gates

Complete the gates in order. A failed gate is a reason to keep the current account-free production build, not to weaken the deletion/privacy requirements.

**Progress (2026-07-14):** Gates 1–3 are reported complete. The next required gate is Gate 4: update the authenticated-build privacy disclosures and Play Console Data Safety form before uploading it to a Play testing track.

### Gate 1 — Confirm the product promise

**Status: complete (reported 2026-07-14).**

**Owner decision:** explicitly choose the next release as the "saved-search sync" release.

- Keep entry points quiet and contextual: Search header `Sign in` changes to a signed-in avatar menu containing Preferences and Search history; History itself can still offer `Sign in to sync`; no account prompts in Refine, Results, or Detail.
- Confirm that synced saved searches work across at least two devices/accounts in the live environment.
- Write one support-ready explanation of what signing in changes, what remains available without an account, and how deletion works.
- Decide whether email verification is required before a user can use sync. Keep this consistent between the Supabase provider settings, app copy, and support explanation.

**Exit criterion:** there is a clear, truthful one-sentence benefit for sign-in, and no enabled account feature depends on a future Price Watch/Deep Dive rollout.

### Gate 2 — Configure a safe authenticated preview build

**Status: complete (reported 2026-07-14).**

Prepare a new EAS preview/internal-test environment without changing the public production release.

- Set the preview build environment variables in EAS (never commit values):
  - `EXPO_PUBLIC_ACCOUNT_UI_ENABLED=true`
  - `EXPO_PUBLIC_SUPABASE_URL`
  - `EXPO_PUBLIC_SUPABASE_ANON_KEY`
  - existing API and Sentry variables
- Keep the two optional-tool flags unset/false.
- In Supabase Auth, confirm email/password and Google are enabled only if each will be offered in the app.
- Add the exact mobile callback generated by `AuthSession.makeRedirectUri({ path: "auth/callback" })` for the signed preview build to Supabase Auth redirect allow-listing. Verify the actual URI from the device/build rather than guessing it.
- Confirm the Android OAuth client/package/signing configuration used by Google matches the preview and eventual Play signing path.
- Build an authenticated preview artifact. `app.config.js` should fail the build if account UI is on but either Supabase public value is missing.

**Exit criterion:** the preview install contains account entry points, can start both enabled auth methods, and contains no accidentally enabled optional tool UI.

### Gate 3 — Live device acceptance testing

**Status: complete (reported 2026-07-14).**

Run this on a physical Android device against deployed Supabase and Render, not only Expo Go or Jest.

1. **Email/password:** create account, sign in, sign out, sign back in, and use password reset if that path is offered.
2. **Google:** finish consent in the browser, return to the app, and confirm the Supabase session survives an app restart.
3. **History sync and preferences:** complete a search while signed out; sign in and verify only migrated local entries are removed locally; then verify saved searches and a changed ranking preference are visible from a second signed-in device. Confirm the saved priority is sent with the next finalize request and changes the shortlist as expected.
4. **Anonymous continuity:** sign out and confirm new searches/history still work locally and search remains completely ungated.
5. **Failure handling:** test cancelled Google consent, bad credentials, offline/auth-network failure, and an expired/revoked session. The app must explain the failure and remain usable for search.
6. **Deletion:** delete the account from Settings, verify the confirmation and signed-out reset, then verify the account's saved searches/auth identity are actually deleted or inaccessible as designed. Also verify the public `https://focamai.com/delete-account` pathway works without the app installed.
7. **Privacy:** inspect the authenticated app's normal flows and Sentry events to confirm the existing scrubbing still avoids auth tokens, email addresses, query text, and request bodies.

Run `npm test -- --runInBand` and `npm run check:android-export` before and after any code correction. Record the build number, test device, date, and failed/blocked cases in the release ticket—not in source-controlled notes if they contain personal account information.

**Exit criterion:** all seven scenarios pass in the live preview build, with no unresolved auth redirect, account migration, or deletion defect.

### Gate 4 — Update the public disclosures before Play submission

**Status: next.**

Update the privacy policy and Play Console to describe the build that will actually be uploaded.

- Update the Privacy screen/web policy only if its current wording does not clearly cover authentication, account-backed saved searches, deletion, and legitimate retained operational/security records.
- In Play Console Data Safety, add **Email address** and **User IDs** as **optional** collected data, used for **Account management**. Mark sharing based on the actual Supabase/service-provider relationship and the current policy guidance; do not copy the old account-free answers unchanged.
- Keep the existing relevant data declarations (search history, user-generated content, audio, app activity, diagnostics, and device IDs) accurate for the same artifact.
- In the Data deletion section, retain the functional public deletion URL and answer the account-creation/deletion questions for an app that now creates accounts.
- Re-check that the in-app privacy link, Play privacy-policy URL, contact address, and public deletion page are all accessible and mutually consistent.

Google Play requires an app that lets users create accounts to provide both a readily discoverable in-app account-deletion path and a web resource for account/data-deletion requests. The deletion path must cover associated account data, while clearly explaining any legitimate retention. [Google Play account deletion guidance](https://support.google.com/googleplay/android-developer/answer/13327111?hl=en)

**Exit criterion:** a reviewer can compare the authenticated build, privacy policy, deletion page, and Data Safety form and find no contradiction.

### Gate 5 — Internal/closed Play validation

- Upload the authenticated build first to the internal or closed testing track—not production.
- Ask testers to cover both sign-in methods, an existing-account sign-in, sign-out, cross-device search-history sync, and deletion.
- Monitor Sentry for auth-route crashes and redirect failures only; preserve the existing privacy-first scrubbing and do not add user-identifying auth telemetry as a shortcut.
- Resolve every release-blocking auth/deletion issue and rerun the affected live checks on a fresh build.

**Exit criterion:** real Play-distributed builds, including the Play signing path, pass the acceptance matrix and no tester is blocked from using core search while signed out.

### Gate 6 — Production release decision

Promote only after Gates 1–5 are complete.

- Rebuild production with account UI and the required Supabase public configuration enabled.
- Reconfirm `EXPO_PUBLIC_PRICE_WATCH_UI_ENABLED` and `EXPO_PUBLIC_DEEP_DIVE_UI_ENABLED` are false/unset.
- Submit the matching, updated Data Safety form with the app update.
- Announce sign-in as saved-search sync plus optional shortlist priorities; do not imply search requires an account.

## Rollback plan

This is a build-time Expo feature flag and Expo Updates is disabled. It cannot be switched off remotely for an already-installed production binary.

If a severe auth, redirect, privacy, or deletion problem appears:

1. Halt promotion and remove the affected track from further rollout where Play Console allows.
2. Build and submit a higher-version replacement with `EXPO_PUBLIC_ACCOUNT_UI_ENABLED=false` and no Supabase public configuration, returning the app to the current account-free experience.
3. Keep the public deletion page and support channel available for any account created during the affected release.
4. Record the cause and fix, then restart from Gate 2 with a new preview build.

## Definition of done

The authenticated production build is live, its Play disclosures match its behavior, users can search without an account, signed-in users can sync saved searches and choose shortlist priorities, and both in-app and web account-deletion paths have been live-tested.
