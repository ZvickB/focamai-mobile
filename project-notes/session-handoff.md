## Current direction

**Partial-shortlist recovery (2026-07-16):** A tester-only first-finalize recovery is implemented. When the shared backend returns `selection.candidateRecovery` (fewer than four credible picks plus a valid distinct suggested query), `ResultsScreen` puts a native, read-only card immediately after the shortlist. `Find better matches` preserves follow-up notes, starts a fresh `cacheMode=refresh` discovery, and navigates through `RetryUpdating`; `Keep these picks` dismisses it. The recovery is stored in the one-hour flow snapshot. `RetryUpdating` now finalizes with existing notes instead of clearing them and does not promise six picks. Mobile analytics records only aggregate shown/accepted/kept data (including count/query length, never query text); the sibling web dev dashboard reports those totals. Focused Jest passed; verify on Expo Go before Play promotion.

**Mobile activity analytics (2026-07-16):** The first mobile funnel slice is implemented. `src/lib/mobileAnalytics.js` is the sole client module; both EAS `preview` and `production` enable it, it uses one random search/session ID pair per search (not a device or account identifier), and silently drops failed tracking requests. `useMobileSearchController` emits search/refinement/results/retry/failure events, while `SearchResultDetailScreen` emits product-open and Amazon-click events. The sibling web backend accepts only the narrow `/api/analytics/mobile` contract and the dev-only web dashboard has a `Mobile` tab. It deliberately does not send typed refinement/retry feedback, raw errors, persistent install IDs, account IDs, or Sentry data. Before uploading a Play build, reconcile the public privacy and Play Data Safety wording with this behavior.

**Mobile account menu and ranking preferences (2026-07-14):** Keep Settings app-level for signed-in users: region plus app/legal/support links. When accounts are enabled, Search changes its signed-out `Sign in` control into the signed-in user's initial/avatar; its native bottom sheet contains email identity, `Preferences`, `Search history`, optional `Price watches`, `Sign out`, and confirmation-protected `Delete account` last. Keep Search history as a Settings fallback only while signed out/account-free so device-local history remains reachable. `Preferences` opens a native five-choice ranking screen (`balanced`, `price`, `lowest_price`, `brand`, `range`) that persists to Supabase `user_preferences` and shapes new finalizations. The account-free Play build still hides all account entry points. The account release promise is saved-search sync plus optional shortlist priorities, so re-run live account-menu, preference persistence, and shortlist QA before its next Play upload.

**Startup splash:** The native Expo splash is wordmark-only on Cleaner Ivory (`#fbf7ef`); do not restore the standalone F mark. `App.js` uses `expo-splash-screen` to keep native artwork visible until the equivalent React Native splash has laid out and painted two compositor frames, with native fading explicitly disabled. Do not wait for `Image.onLoadEnd` here: it can remain pending behind the native splash in an iOS dev client. The React wordmark remains inside a plain native `View`, explicitly sized from `useWindowDimensions()`; root-level absolute fill alone can fall back to the 1024px source bounds and display a cropped, oversized wordmark on iOS cold launch. Reanimated remains confined to the shimmer layer, and the completed splash overlay is removed atomically so its wordmark cannot overlap the navigator. The overlay performs at most one subtle, wordmark-clipped diagonal shimmer after 200ms while genuine font startup remains pending; it never deliberately holds the app after startup is ready, and Reduced Motion always skips it. `assets/splash-wordmark.png` and `assets/splash-wordmark-mask.png` are paired 1024px canvases and must stay geometrically aligned. Development console logs cover JS bootstrap to root/resources/first interactive; device cold/warm timing and native-to-React visual QA remain required.

**Splash adjustment (2026-07-14):** The paired native/React wordmark canvases are 25% larger. Its single wordmark-clipped diagonal shimmer is proportionally wider and modestly stronger. The 200ms conditional shimmer start, two-compositor-frame native handoff, responsiveness-first dismissal, and Reduced Motion behavior remain unchanged.

**Privacy / Play submission:** The 2026-07-02 cross-repo work updated both the mobile Privacy screen and canonical web `/privacy` page and implemented secure self-service account deletion through mobile Settings → Account, authenticated Render `DELETE /api/account`, and public `/delete-account`. `../web` is the Focamai product and brand baseline as well as the product-behavior reference. Mobile should be recognizably aligned with web through local brand assets, Manrope, the teal/orange/warm-neutral family, and calm trusted-assistant voice, but it should not runtime-import from `../web` or treat the web layout as a 1:1 mobile target. Shipping brand assets belong in `mobile/assets/`; the PNG wordmark remains preferred unless branding is intentionally revisited.

Amazon is the current primary commerce path and affiliate target. Mobile UI may name Amazon directly when Amazon is the active source and doing so improves clarity, trust, or conversion. Do not force generic `retailer` language in today's Amazon-first UX, but keep backend/provider logic and normalized product data provider-flexible so another source can be added or swapped later.

Mobile result rows, the selected-result display panel, and detail headings use normalized display titles so Amazon keyword stuffing does not dominate the UI. Raw Amazon/source titles remain in the data and appear behind a quiet full-title disclosure on the detail screen when the display title differs.

**The core staged endpoint foundation is mostly done in code, not full web/mobile behavior parity.** Mobile can call discovery, refinement, finalize, enrichment, query-quality suggestion polling, retry advice, and the follow-up hard-constraint pre-finalize refresh through a thin controller with session and stale-response guards. Mobile now also has temporary explicit marketplace-domain selection for the next search, saves that domain preference with AsyncStorage, asks first-run users to choose a store only after they start a search, and provides a basic Settings screen for changing it later. The first-run choice is a bottom-sheet confirmation that holds the submitted query, saves the selected Amazon store once, then continues that same search into Refine. Mobile intentionally does not plan to use geolocation/GPS marketplace detection; keep explicit store choice plus saved preference unless the user explicitly reopens that product decision. That does not mean mobile acts exactly like the web app under the hood. Analytics, broader persistence, and the full web guided-flow ergonomics are still missing or deferred. The same-candidate-pool retry fallback has been removed from mobile; retry now means asking for a better search query and starting a fresh guided search from that suggestion.

**The next phase is not "just make it prettier."** The user reported the temporary retry UI smoke test and first product-card smoke test are complete, so work is now in mobile-native result/detail UX vertical slices, adding only the product behaviors each slice needs. Mobile UI/UX is expected to deliberately differ from the web layout — preserve product behavior and trust principles, not the exact web layout.

**For UI/UX work, `project-notes/mobile-ui-ux-plan.md` is now the primary implementation-order doc.** It outranks `project-notes/proposed-layout.md` and the PNG mockups for sequencing and screen decisions. `project-notes/mobile-ui-ux-direction.md` remains the principles/taste reference, and the proposed layout plus mockups remain brainstorming inputs only.

**Local repeatable smoke testing is Jest + React Native Testing Library for now.** Maestro files still exist as future Android E2E scaffolding, but the user does not have an Android phone and the emulator is not practical, so do not steer the current workflow through Maestro. Use `npm test` for local component smoke coverage and manual Expo Go for real device/live-backend behavior.

**Retry advice query length is intentionally 80 characters.** The backend clamps suggestions to the shared product-query limit and mobile normalizes them before automatically starting refreshed discovery. Results now routes that retry search through a dedicated read-only `RetryUpdating` screen, which shows the actual improved query and automatically finalizes updated picks. There is still no editable generated-query confirmation step. Do not raise retry advice alone unless the shared limit changes everywhere.

**Auth and history parity:**
- Mobile auth is now more discoverable without becoming a gate: signed-out users see a quiet `Sign in` action in the Search header and a contextual `Sign in to sync` action on Search History. Do not repeat this prompt through Refine, Results, or Detail unless that product decision is intentionally revisited.
- Web has Supabase auth (email/password + Google sign-in via `AuthProvider`/`AuthModal`) and account-backed search history (`saved_searches` table with RLS). Signed-out users keep localStorage history; signed-in users get Supabase-backed history with local-to-remote migration on login. Search remains ungated.
- Mobile now has email/password Supabase auth, Google OAuth, and account-backed history parity for saved searches behind `EXPO_PUBLIC_ACCOUNT_UI_ENABLED=true`. Signed-out users keep device-local `AsyncStorage` history; signed-in users use Supabase `saved_searches`; local entries migrate into the account on sign-in. Mobile stores Supabase session data through `expo-secure-store`, not plaintext `AsyncStorage`, and migration removes only the entries that were actually migrated so sign-out/cancellation mid-sync does not wipe new local history. Google OAuth uses Supabase `signInWithOAuth`, `expo-auth-session`, `expo-web-browser`, the `focamai` app scheme, and the existing SecureStore-backed Supabase session. Supabase permits the production `focamai://auth/callback`, an Expo wildcard, and the current Expo Go callback. Do not use iPad Expo Go to judge OAuth: current Expo guidance says Expo Go cannot reliably test OAuth/OIDC because it cannot use the app's custom callback scheme. Test an installed iOS development or preview build, which uses `focamai://auth/callback`; no temporary callback diagnostic remains in code.
- Do not set up a paid Supabase custom auth domain yet. The default Supabase project domain in the iOS OAuth permission prompt is acceptable while testing/pre-revenue; revisit `auth.focamai.com` once accounts are ready for a polished public rollout and the paid add-on is justified.
- Mobile Prime/delivery parity with web is implemented for the current result surfaces: finalize normalization preserves `delivery` and positive Prime flags, enrichment can upgrade `isPrime`/`delivery` after the initial shortlist, and string-false or negative Prime delivery copy should not show a Prime badge.

**Price Watch and Deep Dive parity:**
- Both optional-tool UIs are hidden by default for the Play release while their implementation remains available behind `EXPO_PUBLIC_PRICE_WATCH_UI_ENABLED=true` and `EXPO_PUBLIC_DEEP_DIVE_UI_ENABLED=true`. Default Product Detail shows neither action and Settings omits Price watches. Routes, watch CRUD, Deep Dive client logic, and the shared backend remain intact; re-enable only after accounts and live flows are deliberately restored and verified.
- When re-enabled, Price Watch requires an ASIN plus positive current price and writes directly to the RLS-protected `price_watches` table; Settings -> Price watches manages the account's five watches. Alerts are email-only, paused watches still count, and `Alert baseline` reflects the backend's post-notification reset. Hiding mobile UI alone does not stop the web scheduler or existing alerts.
- When re-enabled, Deep Dive stays eligibility-controlled and explicit. Mobile polls eligibility quietly, never runs the provider automatically, and opens the dedicated native screen only after user action. The authenticated request has a 60-second cutoff and preserves the web CA-to-US USD fallback.

**What not to do:**
- Do not copy the web/backend content-moderation rule engine into mobile. The shared backend owns the frum-default policy, cache sanitation, and AI-suggestion checks; mobile only preserves returned moderation metadata and renders the native hidden-image state.
- Do not copy the 1552-line web `useGuidedSearch.js` wholesale. That's what broke `main`. The lesson was not "don't build complex things" — it was "don't port complexity before the foundation is proven."
- Do not assume all web data-path features are done. Add missing product behavior in bounded slices when the mobile UX needs it.
- Do not add analytics or new marketplace behavior yet. The earlier "no broader persistence yet" line has been intentionally reopened for one bounded case: see the 2026-07-03 search-flow snapshot persistence entry in `current-status.md`. That is not a general green light for more persistence — treat any further persistence addition as its own product decision.

## Current mobile reality
- Production Sentry is now privacy-conscious Mobile Vitals plus crash reporting. Sentry `8.17.2` uses 10% production tracing, automatic app-start/performance tracing, `reactNavigationIntegration()`, and `Sentry.NavigationContainer`; it records app-start, route, and frame/stall timing only, with HTTP spans disabled so query-bearing URLs, tokens, request/response bodies, and headers are excluded. Events and transactions scrub breadcrumbs, extras, requests, and user context; error messages are redacted while stacks/types remain. Default PII, app hangs, automatic session tracking, profiling, Session Replay, feedback, screen recording, and user-input recording remain disabled. The `preview` EAS profile temporarily sets `EXPO_PUBLIC_SENTRY_DEBUG_ENABLED=true`, giving 100% trace sampling, an `internal` Sentry environment, and a Settings → `Sentry verification (temporary)` route to send a labeled test error/transaction; production explicitly sets this flag false and does not register the verifier route. Build a new signed preview APK, cold-start it, open that route, send both signals, navigate between screens, then confirm Issues, Performance, and Mobile Vitals before removing the temporary verifier. Update Play Data Safety from crash logs only to also declare app-performance diagnostics, using the live Sentry retention/settings as the source of truth.
